### Critical Bug Found: **Sign Error in Integration**

The call decision logic at line 689 is:
```python
if realized_rate < pot_odds + call_margin:
    return -1  # FOLD
```

Higher `call_margin` = harder to call = more folding. Every existing contributor to `call_margin` confirms this semantics:
- `postflop_call_margin()` adds **positive** values for weak hands (lines 574-581)
- `pair_domination_margin()` adds **positive** for dominated pairs (line 582)
- `+0.035` for weak pairs on turn (line 600)
- `+check_resistance`, `+line_strength`, `+nutted_risk` (lines 601-603)
- `+0.04` for weak river hands (line 605)

ALL are additive, ALL increase the fold threshold for weak/vulnerable hands.

But the new integration at line 624 does:
```python
call_margin -= fold_incentive
```

This **subtracts** a positive value, **lowering** the threshold and making it **easier to call** with the very hands the function identifies as needing fold pressure. This inverts the intended behavior — the bot will call *more* with weak holdings facing aggression, which is the opposite of what the "fold incentive" should do.

The correct line should be:
```python
call_margin += fold_incentive
```

### Other Observations
- File sizes: `postflop.py` at 998 lines, `strategy.py` at 993 lines — both dangerously close to the 1000-line limit.
- The `postflop_fold_incentive` function itself is well-designed with proper guards and clamping.
- Only the integration has the sign error.
- No role boundary violations (Algorithmic Logic Architect targeting the assigned files).
- No other correctness issues (no premium hand folding, valid JSON output, no unavailable imports).
