
# 演化流水线瓶颈分析报告

## 1. 概述

### 1.1 流水线总体效率

基于对 6 个代次（v6-v11）的日志分析和代码审计，演化流水线的总体时间效率分布如下：

| 类别 | 占比 | 说明 |
|------|------|------|
| 有效工作 | 25% | LLM 实际产出有价值代码的阶段 |
| 重试浪费 | 42% | Worker 零修改、决策测试失败、Review/Critic 拒绝导致的循环 |
| 等待空闲 | 25% | 多小时间歇性停顿（编排器会话崩溃、守护进程等待等） |
| 固定开销 | 8% | 编译检查、Git 操作、文件 I/O 等 |

**关键结论：仅 1/4 的运行时间在执行有效工作，超过 2/3 的时间被重试和空闲浪费。**

### 1.2 主要发现摘要

1. **编排器多小时间歇停顿是最严重的时间黑洞**：观测到 16 次超过 30 分钟的空隙，最长达 9 小时 17 分钟，原因可能是编排器主循环异常退出后缺乏进程内看门狗自动恢复。
2. **Worker 零修改是最高频的浪费源**：31 次 Worker 产出与父代完全相同的代码，每次浪费 5-15 分钟和 $0.30-$2.00 的 LLM Token 费用。
3. **决策测试 93% 通过率频繁阻塞流水线**：26 次因 1 个关键场景失败（通常是调整了 draw margin 后导致 `river_missed_draw_facing_big_bet` 失败）而阻塞整个流水线。
4. **预提交评估已优化但仍较慢**：`n_games` 已限制为最大 5，但对手间串行执行导致总耗时 300-600 秒。
5. **方向审计误报率偏高**：48% 的调用触发重复警告（32/66），但代码中并无自动重试循环，实际循环由编排器 LLM 的行为决策引起。

### 1.3 关键数据

**代次总时长与成功率：**

| 代次 | 父代 | 总时长 | 结果 | 主要失败次数 |
|------|------|--------|------|-------------|
| v6 | v5 | 12h50m | 手动提交 | 11 次重试 |
| v7 | v5 | 25m34s | 正常提交 | 0 次重试 |
| v8 | v5 | 47h23m | 交叉提交 | 14 次重试 |
| v9 | v8 | 8h20m | 交叉提交 | 10 次重试 |
| v10 | v6 | ~2h | 交叉提交 | 4 次重试 |
| v11 | v10 | 45m39s（进行中） | 进行中 | 1 次重试 |

**成功率：6 个代次中仅 v7 通过标准流水线正常完成（16.7%），4 个通过交叉变异完成，1 个手动提交，1 个仍在进行中。**

## 2. 时间分布分析

### 2.1 各阶段平均耗时占比

| 阶段 | 平均占比 | 典型耗时 | 最大观测耗时 |
|------|---------|---------|-------------|
| Worker 执行+重试 | 35% | 200-500s/次，最多 4 次重试 | v8: 47h（交错进行） |
| 编排器空闲停顿 | 25% | 38min-9h17min | v8: 9h17min |
| 决策测试门失败 | 15% | 30-120s/次，触发 5-20min 重试循环 | v8: 20 次失败 |
| 预提交评估 | 10% | 300-600s/对手，通常 4 个对手 | 2400s（全部对手） |
| Review+Critic | 5% | 60-180s/次 | v8: 9 次调用 |
| 方向审计 | 5% | 30-90s/次 | v8: 22 次警告 |
| 守护进程评估等待 | 3% | 0-600s | 受 `DAEMON_EVAL_TIMEOUT` 限制 |
| 守护进程崩溃+重启 | 2% | 3-120s/重启 | 26 次崩溃 |

### 2.2 串行路径 vs 可并行路径

**严格串行的关键路径：**

```
prepare_generation → direction_audit → master → workers → quality_gates → review → critic → precommit_eval → commit
```

每个阶段的开始都依赖前一阶段的检查点门（checkpoint gate）通过。`tool_gates.py:76-86` 中的 `STAGE_GATE_ALLOWLIST` 定义了硬依赖关系。

**已验证的数据依赖（不可并行化）：**
- `review` 依赖 `quality` 门通过（`tool_gates.py:242`）
- `critic` 依赖 `quality` + `review` 门通过（`tool_gates.py:366`）
- `critic` 的输入包含 `reviewer_feedback`（`tool_gates.py:362`），这是 Reviewer 的输出
- 因此 **Reviewer 和 Critic 不可能并行运行**

**可并行化的路径：**
- Worker 1 和 Worker 2（当目标文件不重叠时）— 当前串行执行（`agent_workers.py:194-211`）
- 预提交评估中多个对手的镜像对战 — 当前串行执行（`tool_eval.py:101-164`）
- Phase 1 中的 Performance Analyst + Match Analyst — 已通过 `asyncio.gather` 并行（`generation_scheduler.py:111-115`）

**死代码发现：** `MAX_PARALLEL_WORKERS=3` 和 `_WORKER_SEMAPHORE` 在 `evolution_infra.py:69,100-108` 中定义，但在 `agent_workers.py` 中从未使用。Worker 始终串行执行。

### 2.3 等待时间 vs 有效工作时间

以 v7（最快成功代次）为理想基线：

| 类别 | v7（理想） | v8（最差） | v6（典型失败） |
|------|-----------|-----------|---------------|
| 有效 LLM 工作 | ~15 min | ~90 min | ~45 min |
| 等待/空闲 | ~2 min | ~30h | ~7h |
| 重试浪费 | 0 min | ~17h | ~5h |
| 总计 | 25m34s | 47h23m | 12h50m |

**v8 的 47 小时中，仅约 90 分钟是有效工作，其余 45.5 小时被重试和空闲浪费。**

## 3. 逐阶段瓶颈分析

### 3.1 Phase 1 — prepare_generation（准备阶段）

| 属性 | 值 |
|------|-----|
| 涉及文件 | `generation_scheduler.py`, `evolution_infra.py`, `agent_master.py` |
| LLM 调用 | 是（Performance Analyst + Match Analyst 并行） |
| 子进程 | 否 |
| 平均耗时 | 30-120s（分析师并行）+ 0-600s（守护进程评估等待） |
| 是否阻塞 | 是（守护进程评估等待阻塞下一代的开始） |

**守护进程评估等待（`wait_for_daemon_eval`）：**
- 轮询间隔：5 秒（`evolution_infra.py:478`）
- 超时上限：600 秒（`DAEMON_EVAL_TIMEOUT`）
- 提前退出：20 场对局 + 评分偏差 < 60（`EVAL_RD_MIN_GAMES=20`, `EVAL_RD_THRESHOLD=60`）
- 标准退出：100 场对局（`MIN_GAMES_FOR_EVAL=100`）
- 降级路径：连续 3 次准备失败后降至 30 场（`orchestrator.py:407-409`）

**必要性评分：8/10** — 可靠的评分需要足够的对战数据，100 场最低门槛确保统计有效性。RD 提前退出优化已减轻了大部分等待。

**跳过风险：** 基于高 RD（评分偏差）的不可靠评分进行源选择，可能导致选择"看似强大但实际因方差偏高"的 bot 作为演化基础。

**优化选项：**

| 选项 | 难度 | 影响 |
|------|------|------|
| 守护进程优先评估最新 bot | 低 | 中 |
| 用 asyncio.Event 替代轮询 | 中 | 低 |
| 在 Phase 2 期间预热线程评估数据 | 高 | 中 |

### 3.2 direction_audit（方向审计）

| 属性 | 值 |
|------|-----|
| 涉及文件 | `direction_auditor.py`, `tool_planning.py` |
| LLM 调用 | 是（单次，无重试循环） |
| 子进程 | 否 |
| 典型耗时 | 30-90s |
| 最长记录 | 1m33s（v9） |

**关键代码行为：** `direction_auditor.py:131-134` 显示方向审计是单次 LLM 调用，**没有代码级别的重试循环**。日志中观测到的多次调用是由编排器 LLM agent 的自主决策触发的，而非代码强制重试。当检测到重复时，约束通过 `tool_planning.py:222-232` 注入到 Master 提示词中。

**必要性评分：5/10** — 重复检测有防止局部最优的价值，但 48% 的警告率表明阈值可能过于敏感。停滞分析器和 Critic 已作为替代的重复检测机制。

**跳过风险：** 演化可能在多个代次中重复相同的方法。但考虑到其他防护机制的存在，风险可控。

**优化选项：**

| 选项 | 难度 | 影响 |
|------|------|------|
| 提高重复检测置信度阈值（要求"高"而非"中"） | 低 | 中 |
| 检测到重复时直接注入约束而非重试 | 低 | 高 |
| 缓存审计结果 15 分钟 | 低 | 低 |

### 3.3 master（Master Architect）

| 属性 | 值 |
|------|-----|
| 涉及文件 | `agent_master.py`, `tool_planning.py` |
| LLM 调用 | 是（最多 `MAX_MASTER_RETRIES=3` 次） |
| 子进程 | 否 |
| 典型耗时 | 60-180s（1-3 次尝试） |
| 最长记录 | v6: 27m31s（含 7h28m 间歇停顿） |

**重试循环（`agent_master.py:53-73`）：** 最多 3 次重试，每次间隔 2 秒。失败条件：LLM 返回的 JSON 不包含 `tasks` 键。每次 LLM 调用无硬性超时。

**边界警告：** Master 有时会将错误的文件分配给 Worker（如 Tuner 被分配编辑 `strategy.py` 而非 `constants.py`）。观测到 9 次边界警告。这些不会阻塞 Master 本身，但会导致下游 Reviewer 拒绝。

**必要性评分：9/10** — Master 是整个演化策略的核心决策者，无法跳过。

**跳过风险：** 无 Worker 任务计划，流水线无法继续。

**优化选项：**

| 选项 | 难度 | 影响 |
|------|------|------|
| 在 Master 提示词中加强文件-角色对应约束 | 低 | 中 |
| 为 Master LLM 调用添加超时（如 300s） | 低 | 低 |

### 3.4 workers（Worker 执行）

| 属性 | 值 |
|------|-----|
| 涉及文件 | `agent_workers.py`, `tool_planning.py` |
| LLM 调用 | 是（每个 Worker 最多 `MAX_WORKER_RETRIES=4` 次，`WORKER_TIMEOUT=1000s`） |
| 子进程 | 是（编译检查、冒烟测试） |
| 典型耗时 | 120-1000s/次尝试 |
| 最长记录 | v8: 47h+（交错进行，53 个 Worker 事件） |

**执行模式：** `_execute_workers()`（`agent_workers.py:166-212`）在 `for` 循环中串行执行所有 Worker 任务。关键行为：**首个 Worker 失败即中止**（`agent_workers.py:210-211`），后续 Worker 不会执行——这实际上节省了失败的 Worker 2 的 LLM 调用成本。

**零修改检测（`agent_workers.py:122-142`）：** Worker 完成后检查目标文件是否与父代相同。如果相同，标记为失败并重试。31 次零修改事件中，v8 有 27 次。

**熔断器（`tool_planning.py:328-336`）：** `MAX_WORKER_FAILURES=6`，但有一个重要缺陷——`failure_count` 按 `len(tasks)` 递增（而非实际失败的任务数），如果 2 个 Worker 中有 1 个失败，计数增加 2，**有效重试预算减半**。

**必要性评分：9/10** — LLM 驱动的代码编辑是系统的核心价值。

**跳过风险：** 无代码变更，整个演化系统停止产出新 bot。

**优化选项：**

| 选项 | 难度 | 影响 |
|------|------|------|
| 目标文件不重叠时并行执行 Worker | 中 | 高 |
| 加强 Worker 提示词以减少零修改率 | 低 | 高 |
| 修复熔断器计数逻辑（按实际失败数递增） | 低 | 中 |
| 预验证 Worker 输出的边界合规性 | 低 | 中 |

### 3.5 quality_gates（质量门）

| 属性 | 值 |
|------|-----|
| 涉及文件 | `tool_gates.py`, `code_verification.py`, `decision_tester.py` |
| LLM 调用 | 否 |
| 子进程 | 是（`py_compile`、冒烟测试、决策测试） |
| 典型耗时 | 30-120s |
| 最长记录 | 取决于决策测试场景数 |

**五项检查（`tool_gates.py:71-85`）：**

1. **编译检查**（`verify_code`）：对每个 `.py` 文件运行 `py_compile`，耗时 1-5s
2. **冒烟测试**（`run_smoke_test`）：运行 1 场镜像对战（2 局游戏），耗时 **60-120s**（注意：是完整子进程镜像对战，非简单的编译检查）
3. **决策测试**（`run_decision_tests_detail`）：每个场景最多 10s 超时，14 个场景，耗时 30-120s
4. **代码大小检查**：检查行数限制，耗时 <1s
5. **代码变更检查**：必须至少有一个 `.py` 文件被修改

**决策测试的实际情况：** `decision_tester.py:26-42` 定义了 12 个 `CRITICAL_SCENARIO_IDS`，`tool_gates.py:78` 的门条件是 `decision_rate >= 0.7 AND critical_ok`（零关键场景失败）。**关键-建议分离机制已经存在**，问题在于 12/14 个场景被归类为关键，使得分离几乎无用。93% 通过率意味着 1 个场景失败，而这个场景很可能是关键的。

**必要性评分：7/10** — 防止灾难性回归（弃牌 AA、过牌坚果牌）至关重要，但当前阈值和分类方式导致了过多的假阻塞。

**跳过风险：** 可能提交灾难性错误的 bot。v8 有 1 次通过 0% 通过率的案例（编译错误 + 全失败），证明质量门的必要性。

**优化选项：**

| 选项 | 难度 | 影响 |
|------|------|------|
| 将部分场景从关键降级为建议（12→8） | 低 | 高 |
| 减少场景超时从 10s 到 5s | 低 | 低 |
| 仅对失败场景进行目标修复而非完整重试 | 中 | 中 |

### 3.6 review（代码审查）

| 属性 | 值 |
|------|-----|
| 涉及文件 | `tool_gates.py`, `agent_review.py` |
| LLM 调用 | 是（`run_claude_query`，无硬性超时） |
| 子进程 | 否 |
| 典型耗时 | 60-180s |
| 最长记录 | v8: 6 min |

**审查内容：** 审查 Worker 产出的 diff，执行角色边界检查，评分 1-10。依赖质量门通过（`tool_gates.py:242`）。最高 3 次重试。

**拒绝原因统计：**
- 角色边界违规（Tuner 编辑 strategy.py 或 Architect 修改 15+ 个数值常量）：最常见
- Worker 零修改：第二常见
- 指令相反（Tuner 做了与 Master 指令相反的操作）

**必要性评分：7/10** — 防止结构损坏和角色违规。v8 有 Worker 做了"与指令完全相反"的操作（减小而非增大下注尺度），Reviewer 成功拦截。

**跳过风险：** 角色违规和结构损坏的代码会被提交。

**优化选项：**

| 选项 | 难度 | 影响 |
|------|------|------|
| 仅重跑被批评的 Worker 而非全部 | 中 | 高 |
| Worker 提示词中加强禁止操作列表 | 低 | 中 |
| 预审查边界检查自动回退违规修改 | 低 | 中 |

### 3.7 critic（Critic 评估）

| 属性 | 值 |
|------|-----|
| 涉及文件 | `tool_gates.py`, `agent_review.py` |
| LLM 调用 | 是（`_run_critic`，无硬性超时） |
| 子进程 | 否 |
| 典型耗时 | 60-180s |

**数据依赖：** Critic 的输入包含 `reviewer_feedback`（`tool_gates.py:362`），因此 **Critic 不能与 Reviewer 并行运行**。这是硬数据依赖。

**必要性评分：7/10** — 独立的战略质量门，防止纯数值调优伪装成结构性变更。

**跳过风险：** 缺乏战略洞察的变更可能通过（如 v8 中 Critic 拒绝了"死代码 + 角色违规"的变更）。

**优化选项：**

| 选项 | 难度 | 影响 |
|------|------|------|
| 根据审查结果选择性重跑 Worker | 中 | 高 |
| 简化 Critic 提示词以加速评审 | 低 | 低 |

### 3.8 precommit_eval（预提交评估）

| 属性 | 值 |
|------|-----|
| 涉及文件 | `tool_eval.py`, `tool_helpers.py` |
| LLM 调用 | 否 |
| 子进程 | 是（`mirror_battle` 子进程调用） |
| 典型耗时 | 300-600s/对手，通常 4 个对手 |
| 最长记录 | 2400s（全部对手串行） |

**对手选择（`tool_helpers.py:399-440`）：** 1 个父代 + 2 个顶级 H2H 对手 + 1 个弱势对手 = 最多 4 个对手。如果是交叉变异，还会加入第二个父代。

**执行方式：** `tool_eval.py:101-164` 中对对手列表进行串行 `for` 循环，每个对手调用一次 `mirror_battle`（内部管理所有 `n_games` 对镜像对）。每次 `mirror_battle` 通过 `loop.run_in_executor` 在线程池中运行，带有 `max(300, n_games*120)s` 的超时。

**已应用的优化：** `n_games` 限制为最大 5（`tool_eval.py:49`），修复了之前 LLM 传入 `n_games=100` 导致 900s 超时的问题（提交 `28d9448`）。

**必要性评分：8/10** — 防止提交在镜像对局中输给父代或在全局对局中退化的 bot。

**跳过风险：** 可能提交比父代更弱的 bot，浪费未来演化周期。

**优化选项：**

| 选项 | 难度 | 影响 |
|------|------|------|
| 对手间并行执行镜像对战 | 中 | 高（节省 60-80% 时间） |
| 父代对决减少 `n_games` 至 1 | 低 | 中 |
| 作为异步后台任务执行 | 高 | 高 |

## 4. 关键等待点深度分析

### 4.1 编排器多小时间歇停顿

**严重程度：极高**

观测到的 16 次超过 30 分钟的空隙：

| 空隙类型 | 次数 | 典型时长 | 最长记录 |
|----------|------|---------|---------|
| 编排器会话崩溃 | 7 | 2-9h | 9h17m（v8: master_done → workers_all_reset） |
| 评估等待 | 5 | 30-60min | 7h28m（v6: master_boundary → prepare） |
| 策略切换间隔 | 4 | 30min-4h | 4h10m（v8: critic_rejected → direction_audit_passed） |

**根因分析：**

`orchestrator.py:176-181` 中的 `CYCLE_TIMEOUT=3600s` 通过 `asyncio.wait_for` 包裹整个 `_stream_response` 协程。经验证，**该超时机制本身是正确的**——如果异步生成器停止产出消息，`async for` 循环会挂起等待，但协程仍在执行中，`asyncio.wait_for` 会在 3600s 后正确触发 `TimeoutError`。

多小时间歇停顿的真正原因是 **编排器主循环本身崩溃**（如 `orchestrator.py:485-497` 中的未处理异常），需要外部进程重启。代码中缺乏进程内看门狗来检测已死的编排器循环。

**会话恢复机制（`orchestrator_session.py:64-148`）** 有 4 种恢复情况（A-D），基于检查点 + 会话状态：
- Case C（检查点 + 会话）：尝试恢复精确的 LLM 对话
- Case D（过期会话清除，需全新开始）：最可能的间歇停顿来源

**缺少的机制：** 没有后台任务监控 `orchestrator_loop` 的活跃状态，无法检测到循环已停止并自动触发恢复。

### 4.2 Worker 执行 + 重试循环

**严重程度：高**

Worker 重试的时间预算分析：

| 组件 | 最坏情况 | 典型情况 |
|------|---------|---------|
| 单次 Worker LLM 调用 | 1000s（`WORKER_TIMEOUT`） | 200-500s |
| 最大重试次数 | 4（`MAX_WORKER_RETRIES`） | 1-2 |
| 2 个 Worker 串行 | 2 × 4 × 1000 = 8000s | 2 × 2 × 400 = 1600s |
| 529 速率限制重试（叠加） | +210s | 0s |
| **单轮总预算** | **8210s（~2.3h）** | **~27min** |

**熔断器缺陷：** `MAX_WORKER_FAILURES=6`（`tool_planning.py:328-336`），但 `failure_count` 按 `len(tasks)` 递增而非实际失败数。2 个 Worker 中有 1 个失败时计数增加 2，**有效重试预算仅 3 轮而非 6 次**。

**零修改浪费详细分析：**

| 代次 | 零修改次数 | 估计浪费时间 | 估计浪费 Token 费用 |
|------|-----------|-------------|-------------------|
| v8 | 27 | 4-7 小时 | $8-$54 |
| v9 | 2 | 10-30 分钟 | $0.60-$4 |
| v6 | 2 | 10-30 分钟 | $0.60-$4 |

### 4.3 预提交评估

**严重程度：中**

当前执行流程（串行）：

```
对手1 (parent)     → mirror_battle(n_games=5) → ~300s
对手2 (top_h2h_1)  → mirror_battle(n_games=5) → ~300s
对手3 (top_h2h_2)  → mirror_battle(n_games=5) → ~300s
对手4 (weak)        → mirror_battle(n_games=5) → ~300s
                                                   总计: ~1200s (20min)
```

如果并行化：

```
所有4个对手同时执行 → ~300s (5min)
节省: ~75%
```

注意：`mirror_battle()` 是按对手调用一次的函数调用，内部管理所有 `n_games` 对镜像对。并行化的单位是 4 个 asyncio 任务（每个对手一个），通过 `asyncio.gather` + `run_in_executor` 实现。

### 4.4 守护进程评估等待

**严重程度：低**

`wait_for_daemon_eval`（`evolution_infra.py:427-482`）的退出条件：

| 条件 | 阈值 | 说明 |
|------|------|------|
| 标准退出 | games >= 100 | `MIN_GAMES_FOR_EVAL` |
| RD 提前退出 | games >= 20 AND rd < 60 | 已实现 |
| 降级退出 | games >= 30 | 连续 3 次失败后 |
| 超时 | 600s | `DAEMON_EVAL_TIMEOUT` |
| 守护进程死亡 | games == 0 AND daemon dead | 立即返回失败 |
| 关机信号 | `shutdown_mgr.is_shutting_down` | 立即退出 |

轮询间隔 5s 在此场景下是合理的。事件驱动通知理论上可行但会增加大量复杂性，收益有限（最多节省几秒轮询延迟）。

## 5. 效率优化建议

### 优先级 1：添加编排器活跃看门狗

**预计节省：** 60-80% 的多小时间歇停顿（每个受影响代次节省 5-40 小时）

**实现难度：** 中

**描述：** 添加后台协程监控流水线阶段转换。如果超过 5 分钟无进展，自动触发恢复（清除会话、基于检查点重启循环）。实现要点：
1. 在 `pipeline_state.json` 中添加 `last_stage_change` 时间戳
2. 在 `orchestrator_loop` 中添加后台任务检查过期状态
3. 强制清除过期会话

预计代码量：约 100 行（`orchestrator.py`）。

**风险评估：** 低风险。看门狗仅在不活跃时触发，正常流水线不受影响。需确保检查点状态的序列化/反序列化健壮。

### 优先级 2：重新分类决策测试场景的关键性

**预计节省：** 50-70% 的质量门重试循环（每个受影响代次节省 15-60 分钟）

**实现难度：** 低

**描述：** 当前 12/14 个场景被归类为 `CRITICAL_SCENARIO_IDS`，使得关键-建议分离几乎无用。将 4-6 个场景从关键降级为建议（如 `flop_flush_draw_facing_cbet`、`turn_two_pair_facing_bet`、`river_missed_draw_facing_big_bet` 等可降级）。无需修改门逻辑——**分离机制已存在于 `tool_gates.py:76-78`**。

具体建议降级的场景：
- `flop_flush_draw_facing_cbet`（听牌跟进不是灾难性错误）
- `river_missed_draw_facing_big_bet`（失败听牌弃牌属正常策略调整）
- `turn_two_pair_facing_bet`（中等牌力处理有策略灵活性）

**风险评估：** 低。保留 8-9 个绝对关键场景（AA/KK/QQ 弃牌、坚果牌过牌等）足以防止灾难性回归。

### 优先级 3：并行化 Worker 执行

**预计节省：** 40-50% 的 Worker 实际运行时间（每代节省 3-8 分钟）

**实现难度：** 中

**描述：** 当 Worker 1 和 Worker 2 的 `target_files` 不重叠时，使用 `asyncio.gather` 并行执行。需要：
1. 添加文件级冲突检测（检查两个 Worker 的 `target_files` 集合是否相交）
2. 修改 `_execute_workers()` 以支持并行路径
3. 更新 `worker_snapshots` 逻辑处理并发访问

当前代码已有目标文件声明机制（`task["target_files"]`），冲突检测直接比较集合即可。

**风险评估：** 中。需确保文件快照逻辑在并发环境下的正确性。建议先实现并行执行 + 冲突检测，在非冲突时退化为串行。

### 优先级 4：并行化预提交评估对手

**预计节省：** 60-80% 的预提交评估时间（每代节省 5-20 分钟）

**实现难度：** 中

**描述：** 将 `tool_eval.py:101-164` 中的串行 `for` 循环改为 `asyncio.gather` + `run_in_executor`，同时运行所有对手的镜像对战。每个 `mirror_battle` 调用是独立的，子进程隔离保证安全。需要重构匹配结果的收集逻辑。

**风险评估：** 低。子进程隔离保证无状态冲突。主要工作是重构结果收集和错误处理逻辑以适应并发模式。

### 优先级 5：减少 Worker 零修改率

**预计节省：** 30-50% 的浪费 Worker 尝试（多日运行中节省 2-8 小时）

**实现难度：** 低

**描述：** 31 次零修改事件中，Worker 产出了与父代完全相同的代码。改进措施：
1. 在 Worker 提示词中添加明确的 "你必须使用 Edit 工具修改代码" 指令
2. 在 Worker 任务中添加 `required_edit_count` 字段，由编排器验证
3. 在 Worker 执行中期（前几次工具调用后）检查文件是否已修改

**风险评估：** 低。仅修改提示词和添加检查逻辑，不改变执行路径。

### 优先级 6：修复熔断器计数逻辑

**预计节省：** 每代节省 0-30 分钟（取决于失败模式）

**实现难度：** 低

**描述：** `tool_planning.py:328-336` 中 `failure_count` 按 `len(tasks)` 递增而非实际失败的任务数。当 2 个 Worker 中有 1 个失败时，计数增加 2，有效重试预算减半。应改为按实际失败的任务数递增。

**风险评估：** 极低。纯逻辑修正。

### 优先级 7：加强 Worker 提示词角色边界约束

**预计节省：** 减少下游 Review 拒绝导致的 5-30 分钟重试循环

**实现难度：** 低

**描述：** 角色边界违规是 Review 拒绝的最常见原因。在 Worker 提示词中：
1. 明确列出禁止操作（如 Tuner 不得添加新函数）
2. 列出允许编辑的文件白名单
3. 添加"在每次 Edit 操作前验证目标文件是否在你的 `target_files` 中"的指令

**风险评估：** 低。纯提示词改进。

### 优先级 8：方向审计简化

**预计节省：** 减少每次警告的 1-2 分钟等待

**实现难度：** 低

**描述：** 当方向审计检测到重复时，直接注入约束到 Master 提示词中，而非等待编排器 LLM 重试。当前代码已在 `tool_planning.py:222-232` 注入约束，但编排器 LLM 有时会选择重新调用审计而非接受结果。可通过在审计结果中明确标注"此为最终结果，请勿重试"来缓解。

**风险评估：** 低。

### 优先级 9：守护进程崩溃根因修复

**预计节省：** 消除累计数小时的评估数据丢失

**实现难度：** 高

**描述：** 26 次崩溃（平均每 2-6 小时一次）的具体原因需要运行时日志分析。建议：
1. 添加结构化错误日志（记录异常堆栈到独立文件）
2. 添加内存监控（`resource` 模块或 `psutil`）
3. 在 `ProcessPoolExecutor` 的 `BrokenProcessPool` 恢复中添加更详细的错误信息

**风险评估：** 中。需要守护进程运行时数据配合分析。

### 优先级 10：冒烟测试超时优化

**预计节省：** 每次质量门检查节省 30-60s

**实现难度：** 低

**描述：** `run_smoke_test`（`code_verification.py:50-61`）运行 1 场完整镜像对战（2 局游戏 × 50 手 = 100 次子进程调用），耗时 60-120s。考虑在 Worker 重试循环中使用轻量级编译检查替代完整冒烟测试，仅在最终质量门（进入 Review 之前）运行完整冒烟测试。

**风险评估：** 中。跳过冒烟测试可能允许运行时错误的 bot 通过 Worker 重试，但这些错误会在最终质量门被捕获。

## 7. 结论

### 7.1 核心发现

1. **流水线仅 25% 时间执行有效工作。** 编排器间歇停顿（25%）和重试浪费（42%）吞噬了 2/3 的运行时间。

2. **编排器会话崩溃是最大时间黑洞。** 9 小时 17 分钟的最长间歇停顿相当于浪费了 18 个完整 v7 代次的时间。添加进程内看门狗是最具成本效益的优化。

3. **决策测试的"关键-建议"分离机制已存在但几乎无用。** 12/14 个场景被标记为关键，导致任何单场景失败都阻塞流水线。重新分类 4-6 个场景为建议级别可立即降低阻塞率。

4. **Worker 零修改是最常见的浪费模式。** 31 次零修改事件浪费了大量 LLM Token 和时间。加强提示词约束是最简单的缓解措施。

5. **标准流水线成功率极低（16.7%）。** 6 个代次中仅 v7 通过标准流水线完成，其余均依赖交叉变异或手动干预。

6. **存在死代码和逻辑缺陷。** `MAX_PARALLEL_WORKERS` 信号量从未在 Worker 路径中使用；熔断器计数逻辑有缺陷。这些应作为代码质量改进修复。

### 7.2 优化优先级总结

| 优先级 | 优化 | 预计节省 | 难度 |
|--------|------|---------|------|
| 1 | 编排器活跃看门狗 | 每代 5-40 小时 | 中 |
| 2 | 重新分类决策测试场景 | 每代 15-60 分钟 | 低 |
| 3 | Worker 并行执行 | 每代 3-8 分钟 | 中 |
| 4 | 预提交评估并行化 | 每代 5-20 分钟 | 中 |
| 5 | Worker 零修改率优化 | 多日运行 2-8 小时 | 低 |
| 6 | 熔断器计数修复 | 每代 0-30 分钟 | 低 |
| 7 | Worker 角色边界约束加强 | 每代 5-30 分钟 | 低 |
| 8 | 方向审计简化 | 每次警告 1-2 分钟 | 低 |
| 9 | 守护进程崩溃修复 | 消除评估数据丢失 | 高 |
| 10 | 冒烟测试策略优化 | 每次质量门 30-60s | 低 |

### 7.3 实施建议

建议按以下顺序实施优化：

**第一阶段（立即可做，低风险高回报）：**
- 优先级 2（决策测试重新分类）：~20 行代码修改
- 优先级 6（熔断器修复）：1 行代码修改
- 优先级 5（Worker 提示词加强）：提示词模板修改
- 优先级 7（角色边界约束）：提示词模板修改

**第二阶段（中等投入，显著回报）：**
- 优先级 1（编排器看门狗）：约 100 行新代码
- 优先级 3（Worker 并行化）：重构 `_execute_workers` 函数
- 优先级 4（预提交评估并行化）：重构对手循环

**第三阶段（长期投入）：**
