### 2. Logic Review — Task 1: River Overbet Function

The function signature is:
```python
def choose_overbet_river(min_raise, my_chips, my_round_bet, to_call, pot, win_rate, value_profile, board_texture, spot_info, opponent_model)
```

**Formula:** `amount = int(to_call + pot_after_call * ratio)` where `pot_after_call = pot + to_call`

When wired from the `to_call == 0` section: `pot_after_call = pot`, so `amount = int(pot * ratio)`. For ratio=1.5 with pot=1000, that's a 1500 raise (1.5x pot overbet). For ratio=2.2, that's 2200. ✅ Mathematically correct.

**Ratio calculation:** `ratio = 1.5 + 0.3 * max(0.0, win_rate - 0.70)`. For win_rate=0.85: ratio=1.545. For win_rate=1.0: ratio=1.59. **Concern:** This is actually quite tight — even with a perfect 1.0 win rate, ratio only reaches 1.59 before position adjustment. The clamp to [1.3, 2.2] means the upper bound is never reached naturally. This is a **minor issue** — the overbet sizing might be too conservative to have meaningful impact vs the existing 1.45 cap, especially with OOP penalty (-0.2) bringing it to 1.3-1.39 OOP. However, this conservatism is also a safety feature.

**Gates are sound:** nut tier only ✅, wetness ≤ 0.35 ✅, pot ≥ 400 ✅, all-in fallback ✅.

### 3. Logic Review — Task 2: choose_raise() Modification

`max_ratio = 2.2 if (allow_river_overbet and round_idx == 3 and value_profile.get("tier") == "nut") else 1.45`

This allows the existing raise logic to scale up to 2.2x pot on the river with nuts, while keeping the 1.45 cap for all other situations. **Backward compatible** (`allow_river_overbet=False` default). ✅

**Interaction concern:** When `to_call == 0`, both `choose_overbet_river()` AND `choose_raise(allow_river_overbet=True)` can trigger. The plan wires overbet FIRST, so if overbet returns a value, choose_raise is never called. If overbet returns None (non-nut tier, wet board, etc.), choose_raise still can't exceed 1.45 (since tier != "nut"). **No double-counting risk.** ✅

### 4. Logic Review — Task 3: thin_cap Changes

**Before:** `thin_cap = 0.46 + 0.08 * wetness + 0.05 * max(0, round_idx - 1)` with `to_call == 0` guard  
**After:** `thin_cap = 0.32 + 0.05 * wetness if round_idx <= 2 else 0.40` without `to_call == 0` guard

**Risk assessment:**
- Removing `to_call == 0` allows thin value raises when facing a bet. This is theoretically correct (thin value raises are valid) but **this is the riskiest change** — if an opponent bets strong and we have a thin value hand, we could raise into a dominated range. The lowered values (0.32 vs 0.46) partially mitigate this.
- The old formula for round_idx=3 (river): `0.46 + 0.08*w + 0.05*2 = 0.56 + 0.08*w`. New: `0.40`. So the river thin cap is significantly lower. ✅ Safer on the river.
- For preflop (round_idx=0): old was `0.46 + 0.08*w`, new is `0.32 + 0.05*w`. Lower across the board. ✅

### 5. Logic Review — Task 4: Wiring in get_action()

Placement: after anti_lock_pressure, before weak_pair_river checks. This means:
1. Anti-lock attacks take priority (defense first)
2. Overbet is attempted for nut hands on river
3. Weak pair river checks follow

This ordering is correct. ✅

Both `choose_raise()` calls get `allow_river_overbet=True`. ✅
